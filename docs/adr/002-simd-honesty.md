# ADR-002: SIMD Honesty and Vectorized Ingestion on Cortex-M0+

## Status
Approved

## Context
High-rate signal processing engines require fast ingestion. In our design, raw data is generated by a simulated ADC. In a production telescope backend, this data would flow from a DMA-driven ADC channel.
We want to optimize data movement throughput. Often, projects claim "SIMD optimization" or "vectorized operations". However, we must be technically honest about our target microcontroller capability:
* **ARM Cortex-M0+ does not have hardware SIMD (Single Instruction Multiple Data)** instructions.
* It lacks the DSP extensions found in Cortex-M4 (e.g., `sadd16`, `ssub16`) and lacks vector extensions like Neon or Helium (MVE).

## Decision
We will explicitly implement **register-width memory bus exploitation** and document it honestly as "pseudo-SIMD" or "bus-width optimization" rather than hardware SIMD.

## Rationale
1. **The 32-bit Bus Width**: The Cortex-M0+ has a 32-bit wide system bus. A standard byte-by-byte memory copy uses byte-store instructions (`STRB`), which write 1 byte per cycle. By packing four 8-bit samples into a single 32-bit word (`u32`) and writing the entire word using a single store instruction (`STR`), we maximize memory bus bandwidth.
2. **Instruction Count Reduction**: Writing 512 bytes requires 512 loop iterations, resulting in 512 load-store instruction pairs. Writing 512 bytes as 32-bit words requires only 128 loop iterations. This reduces loop branch overhead and memory bus transactions by a factor of 4.

## Consequences
* We must align the sample buffer to a 4-byte boundary using `#[repr(C, align(4))]`. Unaligned 32-bit accesses on Cortex-M0+ cause a HardFault error.
* All telemetry and code comments must clearly state that this is register-width loading and NOT ARM DSP or NEON hardware vector instructions.
